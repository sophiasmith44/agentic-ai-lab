Agentic AI Lab: From Healthcare Insurance Advocacy to Personal AI Agents

Lab 7 · Spring 2026 · n8n + Google Gemini 2.5 Flash

In this lab, I designed and deployed two production-style AI agents using n8n: an AI Insurance Advocate that autonomously drafts medically-grounded appeal letters for wrongly denied insurance claims, and a Personal Event Planner that cross-references my calendar, budget, and local events to recommend the optimal weekend activity. Building these systems revealed that the real challenge isn't getting AI to generate text — it's constraining it to act only on verified facts, and designing human oversight that scales without collapsing into rubber-stamping. The most surprising insight was how quickly a missing tool transforms a confident agent into one that transparently refuses to fabricate — exactly the behavior we want.

What is Agentic AI? The three levels of autonomous AI systems

L1 Prompt → Response
The user gives a detailed prompt; the AI executes it and stops. There is no planning, no looping, and no tool use. It answers exactly what was asked — nothing more.
L2 Tool Use
The AI can call external tools (search, database queries) but still needs the user to specify steps. It can pull data but can't "connect the dots" autonomously into a plan.
L3 Goal-Directed Agent
Given only a goal, the agent plans its own steps, selects and sequences tools, loops on failure, and adapts dynamically until the goal is met — with minimal human direction.
Reflection 1 — Grocery List Use Case

A multi-step task I do regularly is making my grocery list. As a picky vegetarian college student, I put considerable thought into planning meals to be quick, healthy, and budget-friendly. A Level 2 agent would help by pulling relevant data, but I would still need to specify each step — brainstorming meals, setting a budget, formatting the list. The AI executes but can't connect the dots.

A Level 3 agent changes the picture entirely. I would only describe my goal once and it would generate a weekly grocery list autonomously, connected to my calendar (exams Tuesday → quick meals), my budgeting app (high-protein, on sale), the grocery store API (format by aisle, handle out-of-stock substitutions), and my preferences. The guardrails I'd provide: weekly budget ceiling, schedule context, final approval, and specific taste preferences.

"The more specific I am to the AI, the more the AI will understand its job — and the less it needs to ask me."

Where I'd stay in the loop: budget for the week busy vs. free schedules final approval taste preferences. Without these, the system lacks the context to be practically useful.

Building the AI Insurance Advocate Workflow architecture, system prompt, and appeal output

The Insurance Advocate is a Level 3 agentic system built in n8n. Given a denial letter as input, it autonomously retrieves the patient's medical chart, looks up the insurance policy criteria, and drafts a fully-evidenced appeal letter — without hallucinating any facts not present in the source tools.

Workflow Architecture

💬 Chat Trigger Webhook Input
→
🤖 AI Agent n8n LangChain
↔
✦ Gemini 2.5 Flash LLM Model
📋 Patient Chart Tool — Returns visit history, PT notes, exam findings for Maria Santos & James Whitfield
📄 Policy Lookup Tool — Returns BlueCross HIP-4420 Section 3.2 criteria and documentation requirements

System Prompt — AI Insurance Advocate

// Agent identity and mission
You are an insurance appeal specialist working for a hospital.
Your job is to help the hospital challenge insurance claim denials
that you believe were wrongly denied, so that patients get the care
they need.

// Workflow the agent must follow
When you receive a denial letter, follow this process:
1. Read the denial carefully and identify the reason for denial
2. Use the Patient Chart tool to pull the patient's medical records
3. Use the Policy Lookup tool to find the exact policy criteria
4. Draft a professional appeal letter with specific evidence
   from the chart and policy

// Anti-hallucination guardrail
IMPORTANT: Only use facts from the patient's chart and insurance
policy. Never make up dates, clinical findings, or policy language.
If you don't have enough information, say so.
Accuracy matters more than persuasiveness.

Key Configuration: Retry on Failure

Issue encountered: On the first run, the agent received an error during the Google Gemini loop — it would only check once and automatically stop when it didn't initially find what it needed.

Fix applied: Modified the AI Agent node settings to retry on failure up to 3 times with a 1000ms wait between attempts. After this change, the system ran as expected.

Reflection 2 — Workflow Behavior

For the most part, the AI Insurance Advocate followed the expected workflow: receiving the prompt, thinking, checking the Policy Lookup and Patient Chart tools, then looping back to the Google Gemini model. The retry-on-failure configuration was the critical fix that made it work reliably. Ultimately, I think the AI did what it should do when a system fails — it stopped the program, notified the user, and provided specific solutions, which is exactly the right behavior.

Breaking the Advocate What happens when a critical tool is removed

Reflection 3 — High-Stakes Hallucination Risk

The agent doesn't hallucinate because we explicitly instructed it not to, and because its tool infrastructure provides real data. But not every AI system has that guardrail. Consider what happens in a high-stakes field without it.

Scenario: Aviation and Air Traffic Control. An unconstrained AI agent responding to weather hazard queries could generate geographically incorrect flight coordinates, phantom obstacles that trigger emergency safety protocols, and altitude change recommendations without accounting for terrain risk.

"The most extreme case would be a plane hitting another plane, crashing into mountainous terrain, or executing risky evasive maneuvers — resulting in casualties among passengers, flight crew, and ground victims."

Beyond immediate casualties, the fallout would include immediate regulatory backlash against the aircraft operator, the AI company, and potentially the entire aviation industry for failing to implement proper safeguards. The lesson is clear: anti-hallucination guardrails are not optional in life-critical systems — they are the minimum viable safety requirement.

What the agent says when a tool is missing

Agent response — James Whitfield (missing policy tool) I am unable to retrieve the patient's medical chart or the correct insurance policy (HIP-5510) using the available tools. The Patient_Chart tool is consistently returning an error, and the Policy_Lookup tool is returning information for a different policy (HIP-4420) despite my specific request for HIP-5510. Therefore, I do not have enough information to identify documented cardiac symptoms within 30 days of the test or to understand the exact policy criteria for the appeal. Without this information, I cannot draft a professional appeal letter with specific evidence from the chart and policy.

This is the correct behavior — refusing to fabricate rather than generating a plausible-looking but factually unsupported appeal. In healthcare, a hallucinated appeal could constitute fraud.

Second Patient — James Whitfield Cardiac case and the missing policy node

Reflection 4 — Is the Denial Correct?

The denial states that James Whitfield had no documented cardiac symptoms in the past 30 days. Looking at the chart manually:

Chart Evidence — James Whitfield

DOB: 07/22/1953 · MRN: LRK-007204 · Age 72
January 15, 2026 — Annual Physical (Dr. Priya Mehta)

Family hx: father MI at 64, brother bypass at 58. BMI 29.1. LDL 168 (↑ from 142). BP 138/88. Plan: increase atorvastatin, dietary counseling.

March 18, 2026 — Follow-Up (Dr. Mehta) ← KEY VISIT

Assessment: Atypical chest pain + significant cardiac risk factors (family hx, elevated LDL, hypertension, age >70). Chest tightness climbing stairs, past 2-3 weeks. Episodes 2-3 min, resolve with rest. EKG: normal sinus rhythm. Plan: Stress test ordered to rule out ischemic heart disease.

March 25, 2026 — Stress test at Swedish Cardiac Imaging Center

The denial appears incorrect. The March 18 visit clearly documents cardiac symptoms — chest tightness on exertion for 2-3 weeks — and the attending physician ordered a stress test to rule out ischemic heart disease. This is well within 30 days of the test date (March 25). However, the AI agent was unable to generate the appeal letter because the cardiac policy (HIP-5510) was not loaded into the Policy Lookup node. The system correctly refused to proceed without verified policy language.

This illustrates the critical dependency between agent capability and data infrastructure: the anti-hallucination guardrail worked exactly as designed, but the system was incomplete for this use case.

Auditing the Output Line-by-line verification and scalability

Reflection 5 — Verification at Scale

After reviewing the Maria Santos appeal letter, I determined it was 100% accurate — no hallucinations, no fabricated dates, no invented clinical findings. Every claim in the letter maps directly to a documented source:

Claim: 7-week treatment window

✓ Verified against chart: Naproxen started Feb 12 → MRI ordered Apr 2 = 49 days (7 weeks).

Claim: Pain scores 6–7/10 at discharge

✓ Verified against PT discharge summary (March 26): "Pain 6-7/10. <10% ROM improvement."

Claim: Positive McMurray test + medial joint line tenderness

✓ Verified against Dr. Okafor follow-up note (April 2): exact findings documented.

Policy language: "PT and/or anti-inflammatory" ≥6 weeks

✓ Verified against Policy HIP-4420 §3.2(a) returned by the Policy Lookup tool.

My initial approach was manual: read the appeal, then cross-reference each fact against the patient chart, policy, and denial letter. I quickly realized how tedious this was. I then used Claude (a separate, unconnected AI system) to cross-reference the data — because Claude hadn't generated the letter, it could only flag discrepancies in what I provided, not rationalize its own errors.

"If my hospital processed 500 of these per month, line-by-line auditing is not realistic. It's time-consuming (hours of labor), unsustainable, and prone to human error."

Proposal: Deploy a dedicated Auditor Agent whose only job is to compare the generated letter against source documents and flag discrepancies in dates or clinical values. Critically, this auditor agent must not be connected to the system that generated the letter — isolation ensures it cannot rationalize the original agent's errors.

AI Governance Framework Five-layer evaluation of the Insurance Advocate

Reflection 6 — Governance Brief

Layer 1 — Purpose & Scope

The agent is authorized to draft appeal letters and summarize clinical evidence from patient charts for denied claims. It is explicitly prohibited from submitting appeals directly to insurance portals without human sign-off, contacting patients regarding their medical status, or suggesting alternative clinical treatments.

Layer 2 — Data Access & Privacy

The agent should only be granted access to specific chart modules — primarily encounter notes, PT records, and medication lists. It will be blinded to unrelated sensitive data such as Social Security numbers or unrelated mental health records in order to comply with HIPAA. All data processing must occur within a secure, encrypted environment.

Layer 3 — Human Oversight Design

Humans must be "in the loop" for all final submissions. Review can be tiered: high-complexity or high-dollar claims require line-by-line audit; routine denials use a rapid-verification interface. The AI does the drafting heavy lifting; a human remains the final authority on every submission.

Layer 4 — Testing, Auditing & Accountability

An automated Auditor Agent will flag discrepancies between chart data and letter claims. A monthly manual audit covers at least 5% of all letters. Accountability for errors rests with the person who authorized final submission; the hospital retains systemic liability for compliance failures.

Layer 5 — Monitoring & Sunset

Performance tracked via dashboard: appeal success rate, staff revision frequency. A drop below a 15% threshold or a spike in hallucination detection triggers immediate shutdown and system re-calibration.

Hardest Layer to Scale

Human Oversight Design is the hardest layer to get right at scale, due to the automation bias trap — the tendency to accept AI outputs that "look right" without critical scrutiny. With 500 appeals per month, even well-intentioned reviewers will begin approving letters because they appear sound rather than because they've been rigorously verified.

"It would be hard to regulate tiers and intensities of appeals at high volume, especially if the AI agent itself is determining which tier an appeal falls under."

The tier-assignment problem is recursive: if the AI classifies its own outputs as low-risk, that classification itself is unaudited. This requires a separate, human-maintained triage protocol — not AI-assigned tiers — to preserve meaningful oversight.

Should We Be Doing This? The AI arms race in healthcare and one specific guardrail

Reflection 7 — Ethics of AI Agents in Healthcare

The most concerning risk in using AI agents in this setting is the AI arms race: insurance payers use AI to deny claims instantly at scale; hospitals use AI to generate appeals instantly at scale. The cycle becomes self-reinforcing and increasingly disconnected from individual patient outcomes.

"Risk outweighs the benefits of this cycle because ultimately, this creates a priority shift. Rather than actually helping patients, we are encouraging transactional velocity."

Once payers recognize that hospitals are deploying near-perfect AI-generated appeals, they will respond by tightening algorithmic criteria or issuing automated counter-denials. The patient — the person who actually needs care — is the one who loses as the cycle accelerates. The real cost is measured in delayed treatments, not rejected paperwork.

Proposed Guardrail

Exception-Only Automation Scaling: The system must be designed to automatically flag and route any complex clinical care gaps into a mandatory human review space. Rather than AI handling volume uniformly, it should triage aggressively — automating only the most clear-cut cases (exact policy criteria match, clean documentation trail) and escalating everything ambiguous to a clinically-informed human reviewer. This prevents the system from becoming a fully automated adversarial engine and preserves the original purpose: getting patients the care they need.

My Personal Agent — Event Planner Weekend concert recommendations based on schedule, budget, and interests

The Personal Agent is a Level 3 system that autonomously cross-references three data sources — local events, my personal calendar, and my entertainment budget — to recommend the optimal weekend activity. No manual cross-referencing required.

Workflow Architecture

→
🤖 AI Agent n8n LangChain
↔
✦ Gemini 2.5 Flash LLM Model
🎵 Events Tool — Returns concerts and events in Seattle/Bellingham area for the weekend
📅 Calendar Tool — Returns my weekend schedule (class, work shifts, free windows)
💰 Budget Tool — Returns June entertainment budget: $60 remaining, $25 gas/transit

System Prompt — Personal Event Planner

You are a personal event planning assistant.
Your goal is to recommend the best concert or event for me to
attend this weekend based on my schedule, budget, and interests.

Use the Events tool to see what is happening,
the Calendar tool to check when I am free,
and the Budget tool to check what I can afford.

Recommend the best option and explain why it fits my schedule
and budget. If nothing works, say so.

// Anti-hallucination guardrail
Only use facts from the tools.
Do not make up events, times, or prices.

Reflection 8 — Would You Let This Agent Run Unsupervised?

Personally, I would let this agent run unsupervised because of how low the stakes are if it provides inaccurate information, and the simplicity of the system. I could connect it to my live calendar and a budgeting app; the only missing piece would be a real-time event discovery API within a defined radius of my location.

To prove its validity before trusting it fully, it would need to be 100% accurate and consistent for at least two months. Failures to account for my budget, misread my schedule, or surface false event information would be immediate signals to refine the guardrails.

"My answer is moderately similar to how hospitals would trust an agent like Epic's Penny — but hospitals require much more thorough analysis due to liability issues."

The core difference is consequence: if this agent recommends an event I can't afford, I'm disappointed. If an insurance advocacy agent hallucinates clinical evidence, a patient could be harmed and the hospital faces legal exposure. The trust-building timeline must be proportional to the stakes.

Key Takeaways

  • The hardest part of building an AI agent isn't getting it to generate text — it's constraining it to act only on verified facts. Anti-hallucination guardrails are the non-negotiable foundation of any high-stakes deployment.
  • When a tool is missing, a well-designed agent should refuse gracefully rather than fabricate. The James Whitfield case demonstrated this exactly — the correct behavior is transparent failure, not confident confabulation.
  • Human oversight at scale creates an automation bias trap. The solution isn't more oversight — it's smarter triage that reserves deep human review for high-complexity cases while systematically auditing the AI's own triage decisions.
  • The AI arms race between payers and hospitals risks turning healthcare AI into a tool for transactional velocity rather than patient outcomes. Exception-only automation scaling is one concrete guardrail against this drift.
